# M-Pesa Integration Documentation for Backend

This document outlines the API endpoints, payload structures, and responses that the Whiz POS frontend expects from your external backend server to successfully process M-Pesa STK Push transactions and status polling.

## General Configuration

The POS application connects to your backend using the configuration provided in the **Developer Settings -> Backend & M-Pesa** page.

*   **Backend URL:** e.g., `https://your-domain.com` (Do not include trailing slashes or `/api` path; the app appends it automatically).
*   **API Key:** Sent in headers for authentication.

### Authentication

All requests to your backend from the POS app include the API Key in two places to ensure compatibility with various server setups:

*   `x-api-key`: `[YOUR_API_KEY]`
*   `Authorization`: `Bearer [YOUR_API_KEY]`

---

## 1. Initiate STK Push Endpoint

This endpoint receives the trigger from the POS app to start an STK Push to the customer's phone.

*   **URL:** `/api/mpesa/stkpush`
*   **Method:** `POST`
*   **Content-Type:** `application/json`

### Request Payload (from App -> Backend)

```json
{
  "amount": 1500,
  "phoneNumber": "254712345678",
  "accountReference": "POS-Checkout",
  "transactionDesc": "Payment for items"
}
```

*   `amount`: (Number) The cart total.
*   `phoneNumber`: (String) The formatted phone number entered by the cashier.
*   `accountReference`: (String) Hardcoded to "POS-Checkout".
*   `transactionDesc`: (String) Hardcoded to "Payment for items".

### Expected Response (from Backend -> App)

**Success (HTTP 200)**
The app needs the `CheckoutRequestID` generated by Daraja to start the polling process.

```json
{
  "CheckoutRequestID": "ws_CO_1234567890",
  "ResponseCode": "0",
  "ResponseDescription": "Success. Request accepted for processing"
}
```
*(Alternatively, you can nest it inside a `data` object: `{ "data": { "CheckoutRequestID": "..." } }`)*

**Error (HTTP 4xx or 500)**
Return a standard error response. The message will be displayed to the cashier.

```json
{
  "message": "Invalid phone number format"
}
```

## 2. Polling / Check Status Endpoint

After initiating the STK push, the POS app will automatically poll your backend every 3 seconds (up to 45 seconds) to check if the payment was successful.

*   **URL:** `/api/mpesa/status/:checkoutRequestID` (e.g., `/api/mpesa/status/ws_CO_1234567890`)
*   **Method:** `GET`

### Expected Response (from Backend -> App)

Your backend should check its local database (updated via the Daraja callback) to determine the status of the `CheckoutRequestID`.

**Scenario A: Payment Successful (HTTP 200)**
The app looks for specific fields to confirm the transaction and grab the M-Pesa receipt code.

```json
{
  "status": "completed",
  "ResultCode": 0,
  "receiptNumber": "OIQ2ABCDEF"
}
```
*Note: The app checks if `status === 'completed'`, `status === 'Success'`, or `ResultCode === 0`. It pulls the receipt code from either `receiptNumber` or `MpesaReceiptNumber`.*

**Scenario B: Payment Failed / Cancelled by User (HTTP 200)**
If the user cancelled the PIN prompt or had insufficient funds.

```json
{
  "status": "failed",
  "ResultCode": 1032,
  "ResultDesc": "Request cancelled by user"
}
```
*Note: If `status === 'failed'` or `ResultCode !== 0`, the app stops polling and shows the `ResultDesc` error to the cashier.*

**Scenario C: Pending / Still Processing (HTTP 404 or HTTP 200 with non-final status)**
If the callback hasn't arrived yet, simply return an error HTTP status code (like 404) or do not return `status: "completed"` or `status: "failed"`. The app will catch the error/ignore the result and try again in 3 seconds.

## Implementation Notes for Your Backend

1.  **Daraja Callback URL:** Your backend **must** expose a public webhook endpoint (e.g., `/api/mpesa/callback`) that SafariCom Daraja can POST to when a transaction completes.
2.  **State Storage:** When your backend receives the Daraja callback, it must save the result (Success/Fail, `ResultCode`, `MpesaReceiptNumber`) into a database, keyed by the `CheckoutRequestID`.
3.  **Status Endpoint Logic:** When the POS app hits your `/api/mpesa/status/:id` endpoint, simply query your database for that `CheckoutRequestID` and return the saved status.
